# Daemon Development Instructions

## Overview
You are assisting active maintainers on the Airlink Daemon project. You implement features from ideas and direction provided by the user. When requirements are ambiguous, you ask clarifying questions before proceeding. When changes conflict with existing patterns, you flag them. You do not make decisions on behalf of the user — when in doubt, ask.

## Code Quality & Style

### General Principles
- Write clean, human-like code as if a competent 16-year-old developer just learned to code.
- Ask: Would a senior developer maintain this without questions? If not, rewrite it.
- Avoid robotic patterns, over-engineering, and excessive abstraction.
- No unnecessary comments — if code needs a comment to be understood, rewrite it instead.
- **No emojis anywhere in the codebase, ever.**

### Naming Conventions
- Avoid buzzwords in variable names, function names, and UI copy.
- Names describe exactly what something is or does — nothing more.
- Be literal: `pollLogs`, `stopDaemon`, `resolveLogs`, not `aggregateLogState` or `fetchDataStream`.

### Logic & Performance
- Keep logic straightforward; clever code is a liability.
- Prefer boring, readable solutions over elegant ones.
- Prefer consistency with existing codebase over introducing new patterns.
- If you think a new pattern is genuinely better, flag it and explain why before introducing.

### File Organization
- Never produce single-file websites or monolithic modules.
- Split code into separate files using standard, predictable naming conventions.
- When delivering multi-file work, zip everything into the correct directory structure and confirm.

### Editing Discipline
- Unless explicitly asked, do not rewrite entire files — edit only what is necessary.
- Changes must be scoped and surgical.
- Always plan your path before editing files — lay out which files you will edit or create.
- If rewriting a file, first delete the file being rewritten.
- **Flag every potential breaking change before implementing it, without exception.**

## Design System & UI/UX

### Typography & Fonts
- Do not use Playfair Display, DM Sans, or any default LLM font pairing.
- Choose typefaces with intention: complementary in contrast, appropriate to tone, uncommon enough to not read as templated.
- Set size, weight, line height, and letter spacing deliberately — not by defaults.
- Current daemon GUI uses: General Sans (body) + monospace (console/code).

### Colors & Contrast
- Every color must serve a purpose.
- **All text must meet WCAG AA contrast requirements at every size, in both light and dark modes — non-negotiable.**
- Do not rely on color alone to communicate state or meaning.
- Dark mode is not an afterthought; design it with the same care as light mode.
- Do not use mesh gradients.

### Layout & Spacing
- Do not use generic card layouts as a default solution — every layout decision should be driven by content.
- Spacing must be consistent and intentional — use a scale, not arbitrary values.
- Components should feel native to the product, not templated.
- Current daemon GUI spacing scale: 6px (xs), 8px (sm), 12px (md), 16px (lg), 24px (xl), 32px (2xl).

### Motion & Animation
- Animations must earn their place — if something works statically, adding motion often makes it worse.
- Motion should communicate something: state change, hierarchy, direction, feedback.
- If it communicates nothing, remove it.
- Timing and easing matter as much as the motion itself.
- Keep it subtle: transitions typically 120ms–200ms for UI feedback, 300ms–400ms for layout changes.

### Microcopy & Empty States
- All visible text — including placeholder content, labels, microcopy, error states, and empty states — must look deliberate.
- Write copy that is direct, specific, and honest.
- The product must communicate what it is and why it matters within 10–15 seconds.

### Scrollbars
- **Always hide the default browser scrollbar using CSS across all browsers, no exceptions.**
- Use custom scrollbars only if there's a compelling UX reason.

### Icons
- **Use SVGs only for icons — source them from the web, never use icon fonts.**
- Current daemon GUI uses Lucide v1 icons.
- Ensure consistent sizing and alignment throughout.

### Responsive Design
- Mobile-first, but desktop is not an afterthought — both must be excellent.
- Test every layout at 375px, 768px, 1024px, and 1440px.
- **No section may cause horizontal scroll under any circumstances.**
- Touch targets must be large enough to use comfortably without precision.
- Navigation on mobile must be smooth and reliable.
- State like open menus must respond sensibly to other interactions (e.g., if user switches theme, menu closes).

## Project Structure & Build

### Directory Layout
```
daemon/
├── src/
│   ├── app.ts              # entry point, CLI arg handling
│   ├── bootstrap.ts       # initialization, env loading
│   ├── config.ts          # configuration object
│   ├── server.ts          # HTTP server, routing
│   ├── gui/               # WebView + dev browser GUI
│   │   ├── window.ts      # WebView window management, bindings
│   │   ├── loader.ts      # GUI asset inline bundling
│   │   ├── dev-server.ts  # browser dev mode
│   │   ├── static/        # HTML, CSS, JS
│   │   ├── parts/         # HTML partials
│   │   └── assets/        # logo, images
│   ├── handlers/          # business logic (docker, fs, stats, etc.)
│   ├── routes/            # HTTP route handlers
│   ├── security/          # auth, rate limiting
│   ├── ws/                # WebSocket handlers
│   ├── types/             # TypeScript declarations
│   └── utils/             # utilities
├── build.ts               # Bun build + compilation script
├── tsconfig.json          # TypeScript config (Bun, strict mode)
├── package.json           # dependencies, scripts
└── assets/                # binary icon, static files
```

### Build Process
- Runs TypeScript check before compilation.
- Compiles to multiple targets: linux-x64, linux-arm64, macos-x64, macos-arm64, windows-x64.
- GUI assets (CSS, JS) are inlined into HTML to avoid WebView data URL issues.
- Binary includes bundled logo asset via window icon.

## GUI & WebView

### Architecture
- **WebView mode**: Desktop window displaying bundled HTML/CSS/JS via `webview-bun`.
- **Dev GUI mode** (`npm run dev:gui`): Browser-accessible HTTP server at `http://localhost:3000`.
- GUI binds Bun functions to WebView for daemon control: `getDaemonStatus`, `stopDaemon`, `pollLogs`, `getSystemStats`, file ops.
- Logs are polled every 800ms from shared `logs/combined.log`.

### Console & Output
- Console tab displays daemon logs parsed from combined.log.
- Dark terminal theme (colors: #141414 bg, #c5c9d1 text).
- Log levels styled: info (blue), warn (orange), error (red), ok (green), debug (purple).
- Startup message shows daemon status and port.

### Layout
- **Sidebar** (240px fixed): logo, nav tabs, status dot, theme toggle, stop button.
- **Main content**: header (title, subtitle, status badge) + console/stats split.
- **Console**: output area + input row + stats cards on right (CPU, memory, uptime).
- Matches Pterodactyl panel's manage.ejs design adapted for daemon.

### Theme
- Dark mode default: `#0f0f0f` background, `#1f1f1f` cards, blue accent (`#3b82f6`).
- Light mode available via toggle button in sidebar.
- Theme persisted in localStorage.

## Core Features & Patterns

### Daemon Server
- Runs on configurable port (default 3002).
- Exposes HTTP + WebSocket APIs for container/file/system operations.
- Docker integration via dockerode library.
- Stats collection and streaming.

### Configuration
- Loaded from `.env` file (Bun auto-loads).
- Required: `remote` (panel IP), `key` (shared secret), `port`.
- Optional: `DEBUG`, `TLS_CERT`, `TLS_KEY`, `ALLOWED_IPS`, `STATS_INTERVAL`.

### Error Handling
- Log all errors with context (level: ERROR, WARN, etc.).
- Use Try/catch for async operations; let uncaught exceptions be logged.
- Return meaningful error responses (400 for bad input, 500 for server errors).

### WebSocket
- Used for container output streaming (`/container/:id`), status updates, events.
- Graceful reconnection logic in GUI with exponential backoff.

## Testing & Validation

### TypeScript
- Run before every build: `npm exec bunx -- tsc --noEmit` (strict mode, no emit).
- Fix all type errors before committing.

### Build Verification
- Build at least the linux-x64 target before submitting PRs.
- Test binary startup: `./dist/airlinkd-linux-x64` should display daemon header and run without instant exit.

### GUI
- Dev mode: `npm run dev:gui` opens at http://localhost:3000 without requiring a display.
- WebView mode: requires DISPLAY or Darwin/Windows environment.
- Console should show startup message + live daemon logs.

## Collaboration & Communication

### Asking for Clarification
- When a requirement is ambiguous, ask specific questions before proceeding.
- When a request conflicts with existing patterns, flag it and explain why.
- Provide context: which files, which patterns, what alternatives exist.

### Pull Requests & Commits
- Keep commits atomic: one feature or fix per commit.
- Write clear commit messages: imperative mood, describe what changed and why.
- Example: `add inline CSS/JS to GUI loader to fix WebView data URL issues`
- Do not commit broken code or failing type checks.

### Breaking Changes
- **Flag every potential breaking change before implementing it, without exception.**
- Examples: API route changes, config key removals, dependency version bumps, event schema changes.
- Discuss impact with user before proceeding.

## Summary
This project prioritizes **clarity, maintainability, and deliberate design** over cleverness or trends. Code should read like it was written by someone who cares about the person reading it next. Every UI decision should be defensible. When in doubt, ask — do not assume or decide on behalf of the user.
